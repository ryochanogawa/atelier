計画に基づいて実装してください。
レポートディレクトリ内のファイルのみを参照してください。

**重要**: 必ず Edit/Write ツールを使ってファイルを変更すること。手動での変更指示やコード提示のみで終わらせないこと。

**重要**: 実装と同時にユニットテストを追加してください。
- 新規作成したクラス・関数にはユニットテストを追加する
- 既存コードを変更した場合は関連テストを更新する
- テストファイルの配置はプロジェクトの規約に従う
- ビルド確認は必須。実装完了後、ビルド（型チェック）を実行し型エラーがないことを確認する
- テスト実行は必須。ビルド成功後、必ずテストを実行し結果を確認する
- 新しい契約文字列（ファイル名、設定キー名等）を導入する場合は、定数として一箇所で定義する

**スコープ宣言（実装開始時に作成）:**
```markdown
# 変更スコープ宣言

## タスク
{タスクの一行要約}

## 変更予定
| 種別 | ファイル |
|------|----------|
| 新規 | `src/example.ts` |
| 変更 | `src/routes.ts` |

## 見積もりサイズ
Small / Medium / Large

## 影響範囲
- {影響するモジュールや機能}
```

**判断ログ（実装完了時、判断を行った場合のみ）:**
```markdown
# 判断ログ

## 1. {判断内容}
- **背景**: {判断が必要になった理由}
- **検討した選択肢**: {選択肢の一覧}
- **根拠**: {選択の理由}
```

**完了前のセルフチェック（必須）:**
ビルド・テスト実行前に以下を確認:
- 新しいパラメータ/フィールドを追加した場合、呼び出し元から実際に渡されていることを grep で確認する
- `??`、`||`、`= defaultValue` の使用箇所で、フォールバックが本当に必要か確認する
- リファクタリング後に置き換え前のコード/エクスポートが残っていないか確認する
- タスク仕様外の機能が追加されていないか確認する
- if/else ブロックが引数の違いだけで同じ関数を呼んでいないか確認する
- 新しいコードが既存の実装パターン（API呼び出しスタイル、型定義スタイル等）と一致しているか確認する
