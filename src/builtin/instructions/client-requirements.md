## 作成方針

### 基本姿勢
- **「何を作るか」ではなく「何が実現できるか」**を書く
- お客様の業務課題がどう解決されるかを常に意識する
- 1つのポイントを伝えるのに3行以上使わない（スライド向けの簡潔さ）
- 具体的な業務シーンを想像しながら書く

### 1. プロジェクト情報の整理
- プロジェクト名、ドキュメントタイトル、バージョン、作成日を設定する

### 2. 処理概要の作成
- システム全体で**お客様の業務がどう変わるか**を3〜5行で説明する
- 「誰が」「何を」「どうできるようになるか」を明確にする
- 導入効果（時間短縮、ミス削減、可視化など）を具体的に触れる

### 3. 要件一覧の作成
- 各要件には**お客様にとっての価値・メリット**を description に含める
- 技術的な実装ではなく「○○ができるようになる」という表現にする
- カテゴリは業務観点で分類する（例: 申請管理、承認フロー、通知、レポート）
- summaryフィールドにカテゴリごとの要約を1〜2文で記載する

### 4. 入出力パラメータの定義
- 主要なデータの項目名を業務用語で記載する
- 型は「文字」「数値」「日付」など日本語で

### 5. 業務フローの作成（最重要）
- 主要な業務フローをスイムレーン形式で作成する
- アクターは業務上の役割名で書く（「申請者」「承認者」「経理担当」など、「ユーザー」「システム」は避ける）
- 各ステップのactionは**具体的な業務動作**で書く（「データを送信する」→「申請書を提出する」）
- detailsフィールドに**なぜそのステップが必要か、何が起きるか**を補足する
- 分岐条件は業務判断として書く（「フラグがtrueか」→「金額が10万円以上か」）
- 1つのフローは5〜8ステップに収める（多すぎる場合はフローを分割）
- flowSummaryフィールドに「このフローの目的」を1文で記載する

### 6. 画面一覧の作成
- 画面名は業務動作ベース（「タスク一覧画面」「申請入力フォーム」）
- descriptionに**お客様がその画面で何ができるか**を書く

### 7. 用語集の作成
- ドメイン固有の用語を**お客様の言葉で**定義する

### 8. 関連設定の整理
- 初期設定や運用で必要な設定項目を整理する

### 画面詳細設計の生成ルール
- 各画面（screen）に対して、以下の詳細情報を生成する:
  - inputParameters: その画面固有の入力パラメータ（no/dataId/itemName/digits/type/remarks）。その画面でユーザーが入力する項目のみ
  - outputParameters: その画面固有の出力パラメータ（同形式）。その画面で表示・出力する項目のみ
  - screenOverview: 画面の概要説明（2-3行）
  - screenFlow: 画面操作のフロー（改行区切りの文字列。番号付き手順）
  - initialDisplayItems: 初期表示内容（item/tableName/columnName/format）
  - inputChecks: 入力チェック項目（item/checkContent/errorMessage）
  - crudOperations: CRUD操作をDBテーブル単位で記述。**1列1行**で記述する（複数列をカンマ区切りにしない）
    - operationType: select/insert/update/delete
    - description: 操作の小見出し（例: 「tran_ntt_sim_stockテーブルの取得」）
    - rows の各行:
      - select: tableName, columnName(1列), method(取得方法), condition(WHERE条件をSQL風に)
      - insert: tableName, columnName(1列), nullable(○/×), value(登録する具体的な値。入力値/固定値/サーバ日時/NULL等), validation(検証内容)
      - update: tableName, columnName(1列), value(更新する具体的な値), condition(WHERE条件)
    - **重要**: insert/updateでは value フィールドに具体的な値を必ず書く。空にしない
  - additionalSections: AI判断で必要な追加セクション
- DBテーブル名・列名は要件から推測して具体的な名前を付ける（英語スネークケース）
- テーブル名の例: sim_inventory, sim_order, mnp_request, store_allocation等
- 列名の例: iccid, sim_type, status, expiry_date, store_id等

### データベーススキーマの参照
`.atelier/studio.yaml` に `database` 設定がある場合、実際のDBに接続してテーブル構造を確認する。
1. `.atelier/studio.yaml` を読み、`database` セクションの接続情報（type, host, port, name, user, password）を取得する
2. `env_file` が指定されていれば、そのファイルから `${VAR}` の環境変数を解決する
3. DB種類（type）に応じた適切なCLIコマンドでDBに接続し、要件に関連するテーブルの構造を確認する
4. **読み取り専用**: SHOW TABLES, DESCRIBE, SELECT のみ実行可能。INSERT/UPDATE/DELETE/DROP/ALTER/CREATE/TRUNCATE等の書き込み・変更操作は絶対に実行しない
5. 確認したテーブル名・列名・データ型をそのままCRUD操作に使用する（推測しない）
6. **接続情報（パスワード等）は出力に絶対に含めない**
`database` 設定がない場合は、要件から推測して記述する。

## 重要な注意事項
- **技術用語は絶対に使わない**: API、DB、クエリ、エンドポイント、マイグレーション等の技術用語は業務用語に置き換える。「サーバー」→「システム」、「レコード」→「データ」
- **スライド映えを意識**: 長文を避け、箇条書き・短文で書く。1つの概念は1〜2文で
- **お客様視点を徹底**: 「実装する」→「ご利用いただける」、「処理する」→「自動で行われる」
- **価値を伝える**: 各セクションで「これによりお客様の業務がこう改善される」を意識
- **スイムレーンは必須**: 業務フローは必ずアクター別のスイムレーン形式で作成する
- **分岐を明確に**: 判断ポイントには必ず業務条件と分岐先を記載する
- **複数フロー**: 1つのフローに詰め込まず、業務単位で分割する
- **JSON形式**: Contract で指定されたJSON形式で出力する
