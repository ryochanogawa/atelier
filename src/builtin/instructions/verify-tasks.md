## 検証手順

### 1. タスク→要件カバレッジ検証
全ての要件が少なくとも1つのタスクでカバーされているか:

| 要件# | 対応タスク | カバー状態 |
|-------|-----------|-----------|
| 1 | Task 1, 3 | ✅ |
| 2 | — | ❌ 未カバー |

### 2. 依存関係の検証
- タスク間の依存関係に循環がないか
- 依存元タスクが依存先より前に配置されているか
- 並列実行可能なタスクが特定されているか

### 3. 各タスクに完了条件（テスト）を定義（CURRANTE方式）
各タスクに対して、実装完了を判定するためのテスト条件を定義する:

```
### Task N: {タスク名}

**完了条件:**
1. [ ] {テストケース1: 正常系}
   - 前提条件: ...
   - 操作: ...
   - 期待結果: ...
2. [ ] {テストケース2: 異常系}
   - 前提条件: ...
   - 操作: ...
   - 期待結果: ...

**テスト種別:** Unit / Integration / E2E
**対象ファイル:** {変更対象のファイルパス}
```

### 4. タスク粒度の検証
- 1タスクが大きすぎないか（目安: 1-3時間、変更ファイル3つ以内）
- 1タスクが小さすぎないか（意味のある成果物があるか）
- タスクの境界が明確か（他タスクとの責務の重複がないか）

### タスク別完了条件
（上記テスト条件形式で全タスク分）

### 検証結果サマリ

| カテゴリ | 件数 | 深刻度 |
|---------|------|--------|
| 未カバー要件 | N | Critical |
| 依存関係問題 | N | Critical |
| 粒度問題 | N | Warning |
| 完了条件不明確 | N | Warning |

### 判定
- **PASS**: 全要件カバー、依存関係正常、完了条件定義済み → 実装開始可能
- **WARN**: 軽微な問題あり → 指摘事項を反映した上で実装開始可能
- **FAIL**: 未カバー要件あり or 依存関係に問題 → タスクの修正が必要

## 原則
- 完了条件は「このテストが通れば実装完了」と判断できる具体性で書く
- テスト条件は要件定義書の受け入れテスト条件（verify-requirements で生成済み）と整合させる
- 実装者が迷わないよう、対象ファイルパスを明記する
