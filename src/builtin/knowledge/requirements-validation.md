# 要件検証知識

## 8カテゴリのチェック観点

要件の抜け漏れを検出するための8カテゴリ。各カテゴリに対してキーワードベースで要件の存在を確認する。

| カテゴリ | チェックキーワード | 不足時の提案 |
|----------|-------------------|-------------|
| エラーハンドリング | エラー、例外、失敗、異常、タイムアウト、リトライ | エラー発生時の挙動、リトライ戦略、ユーザーへの通知方法を定義 |
| セキュリティ | 認証、認可、暗号化、セキュリティ、権限、アクセス制御、脆弱性 | 認証・認可方式、データ暗号化、入力バリデーション、セキュリティ監査を定義 |
| パフォーマンス | 性能、パフォーマンス、レスポンス、スループット、レイテンシ、キャッシュ | 応答時間の目標値、同時接続数、スループット要件を定義 |
| スケーラビリティ | スケール、拡張、スケーラビリティ、負荷、成長 | 将来の利用者数増加、データ量増加への対応方針を定義 |
| データ管理 | バックアップ、復旧、移行、データ保持、アーカイブ、GDPR | データバックアップ方針、復旧手順、データ保持期間、プライバシー要件を定義 |
| 監視・運用 | 監視、モニタリング、ログ、アラート、運用、デプロイ | 監視対象メトリクス、アラート条件、ログ方針、デプロイ戦略を定義 |
| ユーザビリティ | UI、UX、アクセシビリティ、多言語、レスポンシブ、使いやすさ | 対象ユーザー、アクセシビリティ基準、多言語対応、レスポンシブ対応を定義 |
| テスト戦略 | テスト、品質、カバレッジ、QA、受け入れ | テスト戦略、カバレッジ目標、受け入れテスト基準を定義 |

### 判定基準

| 基準 | 判定 |
|------|------|
| 8カテゴリ中3つ以上が未定義 | REJECT - 要件定義が不十分 |
| セキュリティまたはエラーハンドリングが未定義 | Warning - 重要カテゴリの不足 |
| 全カテゴリが網羅されている | OK |

## 矛盾検出パターン

要件間の矛盾を検出するための対立キーワードペア。2つの要件がそれぞれ対立するキーワードを含む場合、矛盾の可能性がある。

| キーワードA | キーワードB | 矛盾の種類 |
|------------|------------|-----------|
| リアルタイム | バッチ処理 | 処理方式の矛盾 |
| オフライン | 常時接続 | 接続要件の矛盾 |
| シンプル | 高機能 | 設計方針の矛盾 |
| 軽量 | フル機能 | スコープの矛盾 |
| 高速 | 高精度 | トレードオフの矛盾 |
| 自動 | 手動承認 | 自動化レベルの矛盾 |
| 匿名 | ユーザー認証 | 認証方針の矛盾 |
| オープン | アクセス制限 | アクセス方針の矛盾 |
| 同期 | 非同期 | 通信方式の矛盾 |
| モノリス | マイクロサービス | アーキテクチャの矛盾 |

### 矛盾の重大度

| 重大度 | 条件 |
|--------|------|
| error | 同一カテゴリで必須（must）と対象外（wont）が混在 |
| warning | キーワードペアによる対立検出 |
| warning | 類似要件（Jaccard係数 > 0.7）で優先度が異なる |

## ギャップ検出の判定基準

### カテゴリ別ギャップ

全要件のテキストを結合し、各カテゴリのキーワードが1つも含まれない場合にギャップとして報告する。

### 受け入れ基準の欠如

| 基準 | 判定 |
|------|------|
| 受け入れ基準が1件も定義されていない要件がある | Gap として報告 |
| 全要件に受け入れ基準がある | OK |

受け入れ基準の抽出パターン:
- 「〜すること」
- 「〜できること」
- 「〜であること」
- 「〜となること」

## 優先度推定（MoSCoW法）

| 優先度 | キーワード |
|--------|-----------|
| Must | 必須、絶対、不可欠、なければならない、しなければならない、必ず、確実に |
| Should | すべき、推奨、重要、望ましい、期待 |
| Could | できれば、オプション、あれば良い、可能であれば、nice to have |
| Won't | 不要、対象外、将来、次フェーズ、スコープ外 |

デフォルト（キーワードなし）の場合は Should として扱う。

## チェックリスト生成

検証後に以下の確認チェックリストを自動生成する:

**必須チェック項目:**
- すべてのステークホルダーが要件に合意しているか
- 要件の優先度は適切に設定されているか
- 必須要件の実現可能性を技術的に検証したか
- テスト戦略は策定したか

**カテゴリ別チェック項目:**
- セキュリティ要件は検討したか
- パフォーマンス要件は定義したか
- 信頼性要件は定義したか
- ユーザビリティ要件は定義したか

**推奨チェック項目:**
- デプロイ・リリース計画は策定したか
- ドキュメント作成計画はあるか
