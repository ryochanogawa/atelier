# テストカバレッジ分析知識

## カバレッジレポートの読み方

### 対応フォーマット

| フォーマット | ファイルパス | 生成ツール |
|-------------|-------------|-----------|
| JSON (coverage-summary) | `coverage/coverage-summary.json` | istanbul, c8 |
| JSON (coverage-final) | `coverage/coverage-final.json` | istanbul, c8 |
| LCOV | `coverage/lcov.info` または `lcov.info` | lcov, c8, jest |

### 3種類のカバレッジメトリクス

| メトリクス | 説明 | 読み方 |
|-----------|------|--------|
| 行カバレッジ (Line Coverage) | テスト実行時に通過した行の割合 | 基本的なカバレッジ指標。未通過行は実行されていないコード |
| ブランチカバレッジ (Branch Coverage) | 条件分岐（if/else, switch等）の各パスが通過した割合 | 行カバレッジより厳密。条件分岐の片側のみテストされている場合を検出 |
| 関数カバレッジ (Function Coverage) | テスト中に呼び出された関数の割合 | 未テスト関数の特定に有効。呼び出されたかどうかのみ判定 |

### LCOV フォーマットの読み方

| プレフィックス | 意味 |
|---------------|------|
| `SF:` | ソースファイルパス |
| `LH:` / `LF:` | テスト通過行数 / 総行数 |
| `BRH:` / `BRF:` | テスト通過ブランチ数 / 総ブランチ数 |
| `FNH:` / `FNF:` | テスト通過関数数 / 総関数数 |
| `DA:行番号,実行回数` | 各行の実行回数（0 = 未カバー） |
| `BRDA:行番号,ブロック,ブランチ,実行回数` | 各ブランチの実行状況（`-` or `0` = 未カバー） |

## カバレッジ不足の判断基準

### ファイル単位の判定

| メトリクス | 目標値 | 警告閾値 | 重大度判定 |
|-----------|--------|---------|-----------|
| 行カバレッジ | 80%以上 | 80%未満 | 50%未満 = high, 50-79% = medium |
| ブランチカバレッジ | 70%以上 | 70%未満 | 40%未満 = high, 40-69% = medium |
| 関数カバレッジ | 全関数カバー | 未テスト関数あり | 4個以上未テスト = high, 1-3個 = medium |

### 全体の判定基準

| 基準 | 判定 |
|------|------|
| 全メトリクスが目標値以上 | OK - カバレッジ十分 |
| high 重大度のギャップが存在 | REJECT - 即座にテスト追加が必要 |
| medium 重大度のみ | Warning - テスト追加を推奨 |
| カバレッジレポートが存在しない | REJECT - テスト実行環境の整備が必要 |

## 優先的にテストすべき箇所の特定方法

### 優先度の決定基準

| 優先度 | 条件 |
|--------|------|
| 高 | ビジネスクリティカルな機能（認証、決済、データ操作） |
| 高 | セキュリティ関連コード（入力バリデーション、認可チェック） |
| 高 | 行カバレッジ 50%未満のファイル |
| 高 | ブランチカバレッジ 40%未満のファイル |
| 高 | 4個以上の関数が未テスト |
| 中 | 行カバレッジ 50-79%のファイル |
| 中 | 主要機能のブランチカバレッジ不足 |
| 低 | ユーティリティ関数、設定値のみのファイル |
| 低 | 行カバレッジ 80%以上だが100%未満 |

### カバレッジ改善のアプローチ

1. **high 重大度のギャップから着手**: 行カバレッジ 50%未満 → ブランチカバレッジ 40%未満の順
2. **未テスト関数の特定**: 関数カバレッジから呼び出されていない関数を洗い出し
3. **未カバー行の集中箇所を特定**: LCOV の `DA:行番号,0` エントリを確認
4. **条件分岐の片側テスト不足**: BRDA で `-` or `0` のブランチを優先

### カバレッジの罠

| アンチパターン | 問題点 |
|--------------|--------|
| カバレッジ100%を目的としたテスト | 意味のないアサーション、保守コスト増大 |
| getter/setter のみのテスト | カバレッジ数値は上がるが品質は向上しない |
| モックだらけのテスト | 実装の変更に脆弱、振る舞いを検証していない |
| 正常系のみのテスト | エッジケース・異常系が未検証のまま |
