# セキュリティ知識

## AI生成コードのセキュリティ課題

AI生成コードには固有の脆弱性パターンがある。

| パターン | リスク | 例 |
|---------|--------|-----|
| もっともらしいが危険なデフォルト値 | 高 | `cors: { origin: '*' }` |
| 古いセキュリティプラクティス | 中 | 非推奨の暗号化、旧式の認証パターン |
| 不完全なバリデーション | 高 | フォーマットは検証するがビジネスルールは未検証 |
| 入力の過信 | 致命的 | 内部APIを常に安全と仮定 |
| コピペ脆弱性 | 高 | 同じ危険なパターンが複数ファイルに繰り返される |

特に注意が必要な箇所:
- 認証/認可ロジック（AIはエッジケースを見逃しやすい）
- 入力バリデーション（構文はチェックするが意味論を見逃す可能性）
- エラーメッセージ（内部情報を漏洩する可能性）
- 設定ファイル（学習データから危険なデフォルトを使用する可能性）

## 優先度解決、オーバーライド、信頼境界

複数の設定/定義ソースの優先度解決、意図的なオーバーライド、拡張ポイント自体は脆弱性ではない。本当の問題は、変更が信頼境界を破るか、より低い信頼レベルの行為者に新しい攻撃能力を与えるかどうか。

| 基準 | 判定 |
|------|------|
| 同一ユーザー・信頼レベル内で文書化された優先度ルールに従う | OK |
| 明示的なセレクタや引数が対象を選択し、文書化された優先度モデルに従う | OK |
| 高優先度の定義が低優先度に勝つが、文書化されたカスタマイズ契約内で権限やデータアクセスを拡大しない | 最大でも Warning |
| 低信頼の行為者が高信頼の設定をオーバーライドし、コード実行や権限昇格が可能に | REJECT |

## インジェクション攻撃

**SQLインジェクション:**

- 文字列連結によるSQL構築 → REJECT
- パラメータ化クエリの不使用 → REJECT
- ORMのrawクエリでのサニタイズ不足 → REJECT

```typescript
// NG
db.query(`SELECT * FROM users WHERE id = ${userId}`)

// OK
db.query('SELECT * FROM users WHERE id = ?', [userId])
```

**コマンドインジェクション:**

- `exec()`, `spawn()` での未検証入力 → REJECT
- シェルコマンド構築での不十分なエスケープ → REJECT

```typescript
// NG
exec(`ls ${userInput}`)

// OK
execFile('ls', [sanitizedInput])
```

**XSS（クロスサイトスクリプティング）:**

- HTML/JS へのエスケープなしの出力 → REJECT
- `innerHTML`, `dangerouslySetInnerHTML` の不適切な使用 → REJECT
- URLパラメータの直接埋め込み → REJECT

## 認証と認可

**認証の問題:**

| 基準 | 判定 |
|------|------|
| ハードコードされた認証情報 | 即座に REJECT |
| 平文パスワードの保存 | 即座に REJECT |
| 弱いハッシュアルゴリズム（MD5, SHA1） | REJECT |
| 不適切なセッショントークン管理 | REJECT |

**認可の問題:**

| 基準 | 判定 |
|------|------|
| 権限チェックの欠如 | REJECT |
| IDOR（安全でない直接オブジェクト参照） | REJECT |
| 権限昇格の可能性 | REJECT |

```typescript
// NG - 権限チェックなし
app.get('/user/:id', (req, res) => {
  return db.getUser(req.params.id)
})

// OK
app.get('/user/:id', authorize('read:user'), (req, res) => {
  if (req.user.id !== req.params.id && !req.user.isAdmin) {
    return res.status(403).send('Forbidden')
  }
  return db.getUser(req.params.id)
})
```

## データ保護

**機密情報の漏洩:**

| 基準 | 判定 |
|------|------|
| ハードコードされた API キー、シークレット | 即座に REJECT |
| ログ内の機密情報 | REJECT |
| エラーメッセージでの内部情報漏洩 | REJECT |
| コミットされた `.env` ファイル | REJECT |

**データバリデーション:**

| 基準 | 判定 |
|------|------|
| 未検証の入力値 | REJECT |
| 型チェックの欠如 | REJECT |
| サイズ制限の未設定 | REJECT |

## ロギングとマスキング

ログやレスポンスへの機密情報漏洩を防止する。

**ログに記録してはいけない情報:**
- パスワード、トークン、APIキー
- クレジットカード番号、個人識別番号
- セッションID、認証ヘッダー値
- 個人情報（メール、電話番号）デバッグ目的で必要な場合を除く

```typescript
// NG - パスワードがログに記録
logger.info('User login attempt', { email, password })

// OK - 機密フィールドを除外
logger.info('User login attempt', { email })
```

| 基準 | 判定 |
|------|------|
| ログにパスワード、トークン、APIキーが含まれる | REJECT |
| エラーレスポンスにスタックトレースや内部パスが含まれる | REJECT |
| toString() が機密フィールドを公開 | REJECT |
| ログレベルに関係なく機密情報が出力可能 | REJECT |
| デバッグログにPIIが含まれるが本番では無効 | Warning。設定ミスのリスク |

## 暗号化

| 基準 | 判定 |
|------|------|
| 弱い暗号アルゴリズムの使用 | REJECT |
| 固定 IV/Nonce の使用 | REJECT |
| ハードコードされた暗号鍵 | 即座に REJECT |
| HTTPS 未使用（本番環境） | REJECT |

## ファイル操作

**パストラバーサル:**

- ユーザー入力を含むファイルパス → REJECT
- `../` のサニタイズ不足 → REJECT

```typescript
// NG
const filePath = path.join(baseDir, userInput)
fs.readFile(filePath)

// OK
const safePath = path.resolve(baseDir, userInput)
if (!safePath.startsWith(path.resolve(baseDir))) {
  throw new Error('Invalid path')
}
```

## OWASP Top 10 チェックリスト

| # | リスク | 主な対策 |
|---|--------|---------|
| A01 | アクセス制御の不備 | 最小権限原則、デフォルト拒否 |
| A02 | 暗号化の失敗 | 強力なアルゴリズム、適切な鍵管理 |
| A03 | インジェクション | パラメータ化クエリ、入力サニタイズ |
| A04 | 安全でない設計 | 脅威モデリング、セキュアデザインパターン |
| A05 | セキュリティの設定ミス | ハードニング、不要機能の無効化 |
| A06 | 脆弱で古いコンポーネント | 依存関係の定期的な更新、脆弱性スキャン |
| A07 | 識別と認証の失敗 | MFA、パスワードポリシー |
| A08 | ソフトウェアとデータの整合性の失敗 | CI/CD パイプラインの保護 |
| A09 | セキュリティログとモニタリングの失敗 | 包括的なログ記録、アラート設定 |
| A10 | SSRF | URL検証、ネットワーク分離 |
