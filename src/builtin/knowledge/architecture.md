# アーキテクチャ知識

## ファイル構成と設計

**ファイルサイズ基準:**

| 基準 | 判定 |
|------|------|
| 単一ファイル 200行超 | 分割を検討 |
| 単一ファイル 300行超 | REJECT |
| 単一ファイルに複数の責務 | REJECT |
| 無関係なコードの同居 | REJECT |

**モジュール構造:**

- 高凝集: 関連する機能をまとめる
- 低結合: モジュール間の依存を最小化
- 循環依存の禁止
- 適切なディレクトリ階層

**操作の発見容易性:**

同じ汎用関数が異なる目的でコードベース全体に散在していると、すべての呼び出し箇所を grep しない限りシステムの全体像を把握できない。関連する操作は目的別の関数として単一モジュールにまとめる。そのモジュールを読むだけでシステムが行う操作の全一覧が把握できるようにする。

| 判定 | 基準 |
|------|------|
| REJECT | 同じ汎用関数が異なる目的で3箇所以上から直接呼ばれている |
| REJECT | 全操作の把握にすべての呼び出し箇所の grep が必要 |
| OK | 目的別の関数が定義され、単一モジュールにまとめられている |

**公開API面:**

公開APIはドメインレベルの関数と型のみを公開する。インフラ内部（プロバイダ固有の関数、内部パーサなど）はエクスポートしない。

| 判定 | 基準 |
|------|------|
| REJECT | インフラ層の関数が公開APIからエクスポートされている |
| REJECT | 内部実装の関数が外部から呼び出し可能 |
| OK | 外部はドメインレベルの抽象のみを通じてやり取りする |

**関数設計:**

- 1関数1責務
- 30行超の関数は分割を検討
- 副作用を明確に定義

**レイヤー設計:**

- 依存方向: 上位層 → 下位層（逆方向は禁止）
- Controller → Service → Repository のフローを維持
- 1インタフェース = 1責務（巨大な Service クラスを作らない）

## ディレクトリ構造パターン

構造パターンの選択:

| パターン | ユースケース | 例 |
|----------|------------|-----|
| レイヤード | 小規模、CRUD中心 | `controllers/`, `services/`, `repositories/` |
| バーティカルスライス | 中〜大規模、機能独立性が高い | `features/auth/`, `features/order/` |
| ハイブリッド | 共通基盤 + 機能モジュール | `core/` + `features/` |

バーティカルスライスアーキテクチャ（機能単位のコード配置）:

```
src/
├── features/
│   ├── auth/
│   │   ├── LoginCommand.ts
│   │   ├── LoginHandler.ts
│   │   ├── AuthRepository.ts
│   │   └── auth.test.ts
│   └── order/
│       ├── CreateOrderCommand.ts
│       ├── CreateOrderHandler.ts
│       └── ...
└── shared/           # 機能横断の共通コード
    ├── database/
    └── middleware/
```

バーティカルスライスの判定基準:

| 基準 | 判定 |
|------|------|
| 単一機能が3層以上に跨る | スライスを検討 |
| 機能間の依存が最小 | スライス推奨 |
| 50%以上が共通処理 | レイヤードを維持 |
| チームが機能単位で編成 | スライス必須 |

禁止パターン:

| パターン | 問題点 |
|----------|--------|
| 肥大化した `utils/` | 責務不明の墓場になる |
| 安易な `common/` 配置 | 依存関係が不明確に |
| 過度なネスト（4階層以上） | ナビゲーション困難 |
| 機能とレイヤーの混在 | `features/services/` は禁止 |

## 関心の分離

- 読み取りと書き込みの責務を分離
- データ取得はルート（View/Controller）で行い、子に渡す
- エラーハンドリングの集約（try-catch の散在禁止）
- ビジネスロジックが Controller/View に漏れない

## コード品質の検出基準

**説明的コメント（What/How）の検出:**

コードの挙動をそのまま自然言語で繰り返すコメントを検出する。

| 判定 | 基準 |
|------|------|
| REJECT | コードの挙動を自然言語で復唱している |
| REJECT | 関数名/変数名から自明なことを繰り返している |
| REJECT | 関数名を言い換えただけで情報を追加しない JSDoc |
| OK | 特定の実装を選んだ理由を説明 |
| OK | 一見不自然な挙動の理由を説明 |
| Best | コメント不要 — コード自体が意図を伝えている |

```typescript
// REJECT - コードの復唱（What）
// 中断された場合、即座に中止する
if (status === 'interrupted') {
  return ABORT_STEP;
}

// OK - 設計判断（Why）
// ユーザーの中断はストローク定義のトランジションより優先する
if (status === 'interrupted') {
  return ABORT_STEP;
}
```

**直接的な状態変更の検出:**

配列やオブジェクトの直接変更を検出する。

```typescript
// REJECT - 配列の直接変更
const steps: Step[] = getSteps();
steps.push(newStep);           // 元の配列を変更
steps.splice(index, 1);       // 元の配列を変更

// OK - イミュータブル操作
const withNew = [...steps, newStep];
const without = steps.filter((_, i) => i !== index);
```

## アンチパターン検出

以下のパターンを検出した場合は REJECT:

| アンチパターン | 問題点 |
|--------------|--------|
| God Class | 単一クラスに責務が集中し変更影響が巨大 |
| Shotgun Surgery | 1つの変更が多数のファイルに波及 |
| Feature Envy | 他クラスのデータを過度に参照 |
| Primitive Obsession | ドメイン概念をプリミティブ型で表現 |
| 循環依存 | モジュール間の相互参照 |
| Leaky Abstraction | 抽象の裏の実装詳細が漏洩 |
