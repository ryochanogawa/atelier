# テストギャップ検出知識

## テストがないファイル/関数の特定方法

### カバレッジギャップの3種類

| ギャップ種類 | 検出方法 | 説明 |
|-------------|---------|------|
| function | 関数カバレッジで `functionsTotal > functionsCovered` | テスト中に一度も呼び出されていない関数 |
| branch | ブランチカバレッジで `branchRate < 0.7` | 条件分岐の一部パスが未通過 |
| line | 行カバレッジで `lineRate < 0.8` | 実行されていないコード行が多い |

### 未テストファイルの判定

| 条件 | 判定 |
|------|------|
| カバレッジレポートにファイルが存在しない | 完全未テスト - テストファイルの作成が必要 |
| 行カバレッジ 0% | テストファイルは存在するが実行されていない可能性 |
| 関数カバレッジ 0% | テストファイルが他ファイルの関数のみテストしている可能性 |

### 未テスト関数の特定手順

1. カバレッジレポートからファイル別の `functionsCovered` と `functionsTotal` を取得
2. `functionsTotal - functionsCovered` が未テスト関数の数
3. LCOV の `FN:行番号,関数名` と `FNDA:実行回数,関数名` を照合して具体的な関数名を特定
4. `FNDA:0,関数名` のエントリが未テスト関数

## エッジケースの洗い出し方法

### 汎用的なエッジケースパターン

| カテゴリ | エッジケース例 |
|---------|-------------|
| 境界値 | 0、1、最大値、最小値、境界値+1、境界値-1 |
| 空値 | null、undefined、空文字列、空配列、空オブジェクト |
| 型の境界 | 整数の最大値/最小値、浮動小数点の精度限界 |
| 文字列 | マルチバイト文字、特殊文字、非常に長い文字列、改行を含む文字列 |
| コレクション | 空コレクション、1要素、重複要素、大量要素 |
| 非同期 | タイムアウト、同時実行、競合状態 |
| エラー | ネットワークエラー、パーミッションエラー、ディスクフル |

### ブランチカバレッジベースのエッジケース検出

条件分岐の未カバーブランチからエッジケースを推定する:

| 未カバーパターン | 推定されるエッジケース |
|----------------|---------------------|
| if の else 側が未カバー | 条件が false になるケース（異常系、デフォルト値） |
| switch の default が未カバー | 想定外の値が渡されるケース |
| catch ブロックが未カバー | 例外が発生するケース |
| 三項演算子の片側が未カバー | 条件の反対側のケース |
| 早期リターンの後が未カバー | ガード条件を通過するケース |

## テストケース提案のパターン

### ギャップ種類別のテストスケルトン

**未テスト関数 (function ギャップ):**

```typescript
describe("対象関数名", () => {
  it("正常系: 期待される結果を返す", () => {
    // Arrange - テストの前提条件
    // Act - テスト対象を実行
    // Assert - 結果を検証
  });

  it("異常系: エラーを適切に処理する", () => {
    // Arrange - エラーを発生させる条件
    // Act & Assert - 例外やエラー応答を検証
  });
});
```

**未カバーブランチ (branch ギャップ):**

```typescript
describe("対象関数名 - ブランチカバレッジ", () => {
  it("条件分岐: trueの場合", () => {
    // 条件が true になる入力でテスト
  });

  it("条件分岐: falseの場合", () => {
    // 条件が false になる入力でテスト
  });
});
```

**未カバー行 (line ギャップ):**

```typescript
describe("対象関数名 - 行カバレッジ改善", () => {
  it("未カバー行を通過するテスト", () => {
    // 未カバー行を実行するための入力条件を特定
    // その条件でテスト対象を実行
  });
});
```

### テスト提案の優先度

| 優先度 | ギャップの重大度条件 |
|--------|-------------------|
| high | 行カバレッジ 50%未満、ブランチカバレッジ 40%未満、未テスト関数 4個以上 |
| medium | 行カバレッジ 50-79%、ブランチカバレッジ 40-69%、未テスト関数 1-3個 |
| low | それ以外 |

### E2Eテストシナリオの生成パターン

要件カテゴリに応じた E2E テストシナリオを生成する:

| 要件カテゴリ | シナリオ種別 | ステップ構成 |
|-------------|-----------|-------------|
| feature | 正常系 | セットアップ → 機能実行 → 受け入れ基準検証 |
| security | セキュリティ検証 | 不正入力試行 → 権限なし操作試行 → エラーメッセージ確認 |
| performance | パフォーマンス検証 | 通常負荷テスト → 高負荷テスト |

### テスト命名規則

| パターン | 例 |
|---------|-----|
| 正常系 | `正常系: 有効な入力で期待される結果を返す` |
| 異常系 | `異常系: 無効な入力でエラーを返す` |
| 境界値 | `境界値: 最大値で正常に処理される` |
| エッジケース | `エッジケース: 空配列で空結果を返す` |
